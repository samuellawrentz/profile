## Voice & Tone

- **Conversational** â€” writes like talking to a friend. Uses "I", "you", "we" freely.
- **Genuinely enthusiastic** â€” real excitement, not performed hype. "How cool is that?" / "I love React, and so do you!" / "Man!, he was on fire."
- **Self-deprecating and honest** â€” admits laziness, procrastination, boredom openly. "dedication sucks" / "I went dark for the next two years" / "If you are lazy even to do that click..."
- **Opinionated** â€” takes a clear stance. "I'm not going back." / "Forget about margins." / "Gone are the days of class-based components."
- **Direct** â€” contractions, informal phrasing. "kinda", "stuff", "WTH is `minmax()`?"
- **Organic pop culture** â€” movie/game references when they fit naturally, not forced. "I felt like thanos, having the powers of the stone" / "May the force be with you"
- **Raw over polished** â€” slight rough edges are fine. The early posts had typos ("procastination", "collegue") and that felt human.

## Structure Patterns

- **Personal anecdote or opinion as intro** â€” jump in with a real story or strong take. 1-2 paragraphs max. Never "Hey there tech fam!" or "Buckle up!"
- **Numbered lists for personal journeys** â€” the blog-about-blog and salvaged-website posts use numbered timelines. This is a signature pattern.
- **H2 headers as conversational statements** â€” "Okay, but How?", "Finally, some code", "Do we need media queries? I guess not.", "Why Minimal?"
- **Code first, explain after** â€” show the snippet, then break it down piece by piece
- **Aside banners** for tips: `<aside class="banner banner-info">...</aside>`
- **Embedded demos** when applicable â€” CodeSandbox iframes
- **One short closing** â€” 1-2 sentences max. "Hope this helps" / "Happy coding!" / one italicized sentence. Never multiple closing sections (no "Bottom Line" + "Quick Tips" + "Last Word" + P.S. chains)
- **Short to medium length** â€” 300-700 words for most posts. The bun-modules post (technical deep-dive) is ~900 words and that's the upper bound. Never 1200+ word bloat.

- "Let me break it down for you" / "Here's the thing" â€” filler
- "Keep coding, keep vibing" as a sign-off repeated across posts

- Multiple closing sections (Bottom Line + Tips + Last Word + P.S.)

### General
- "In this article, we will explore..." / "In conclusion" / "without further ado"
- "leverage", "utilize", "comprehensive", "robust", "streamline", "delve", "demystifying", "revolutionizing", "game changer"
- "it's important to note", "let's dive in", "let me tell you"
- Textbook tone â€” bullet-point walls without personality
- Over-explaining basics â€” trust the reader
- Excessive self-links shoehorned into unrelated sentences (some older posts did this too aggressively)

## Code Examples

- Keep snippets short and focused (5-15 lines ideal, up to 20 for real-world examples)
- Inline comments sparingly, only for the non-obvious
- Show the simplest working version first, then the advanced pattern
- Use `//bad` and `//good` labels when contrasting approaches
- Real code from real projects > contrived examples
- Comparison tables work well for benchmarks (see bun-modules post)

## Hero Images

- Each post has a custom hero image in the same directory as the MDX file
- Referenced via relative path in frontmatter: `heroImage: ./image-name.png`
- Images are typically descriptive/thematic (e.g., cheetah for "faster builds", grid illustration for CSS grids)
- New posts should use `header.png` as the filename (this feeds OG image generation)

## OG Image Generation

- The prebuild script (`scripts/generate-og-images.cjs`) looks for **`header.png`** (exact name) in the post directory
- If found: uses it as a full-bleed background (1200x630), overlays a dark angled gradient + title in white bold 70px Courier + logo
- If missing: falls back to a plain dark gradient background with the same text
- The `heroImage` frontmatter field is NOT used for OG â€” only `header.png` matters
- To get a custom OG background: place a `header.png` in the post directory (any aspect ratio works, it scales to cover)
